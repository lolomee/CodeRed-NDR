2. Access Model & Credentials

User Accounts

UserAuth MethodShellAccess LevelPurpose
ubuntuSSH key (PEM) onlyFull bash + sudoAdmin (you)Full system access, maintenance, debugging
coderedaiPassword (coderedai)Restricted CLI (cli.py)CustomerSetup wizard, status, basic config changes
rootDisablednologin (no SSH)NoneLocked — use ubuntu + sudo instead

SSH Configuration

# sshd_config enforces:
PermitRootLogin no
AllowUsers ubuntu coderedai

# ubuntu = key-only (no password SSH)
Match User ubuntu
    PasswordAuthentication no
    PubkeyAuthentication yes

# coderedai = password-only (no key upload escape)
Match User coderedai
    PasswordAuthentication yes
    PubkeyAuthentication no
    ForceCommand /usr/bin/python3 /opt/codered/shell/cli.py

How to SSH In

# As admin (full access):
ssh -i NDR.pem ubuntu@<sensor-ip>

# As customer would (restricted CLI):
ssh coderedai@<sensor-ip>
# Password: coderedai (default)
⚠ NEVER share the PEM key or ubuntu account with customers

The customer should only know about the coderedai account. The ubuntu account is your administrative backdoor for maintenance and troubleshooting.

Customer Sudoers (Limited)

The coderedai user can only run these commands via sudo (defined in /etc/sudoers.d/codered):

# Allowed commands (NOPASSWD):
hostnamectl, nmcli, ip, ethtool         # Network config
so-status, so-restart, so-*-restart      # SO service management
salt-call --local                        # Apply Salt states
systemctl is-active/restart              # Service status
docker ps                                # Container status
shutdown                                 # Reboot/shutdown
timedatectl, journalctl, pgrep           # Diagnostics
chpasswd                                 # Password change
cp/chmod/mkdir for /etc/codered/         # Config file writes

Applying OS Security Patches

# SSH as admin
ssh -i NDR.pem ubuntu@<sensor-ip>

# Check available updates
sudo dnf check-update        # Oracle Linux / RHEL
sudo apt list --upgradable   # Ubuntu / Debian

# Apply security patches only
sudo dnf update --security -y        # Oracle Linux
sudo apt-get upgrade -y              # Ubuntu

# If kernel was updated, reboot
sudo reboot
⚠ Notify customer before rebooting

OS patches that require a reboot will cause a brief monitoring gap. Schedule during a maintenance window and notify the customer.

Updating Security Onion

# SSH as admin
ssh -i NDR.pem ubuntu@<sensor-ip>

# Check current SO version
cat /etc/soversion

# Run SO update (soup = Security Onion UPdate)
sudo soup

# This will:
# - Pull latest Docker images for Zeek, Suricata, etc.
# - Update SO Salt states
# - Restart affected containers
# Takes 10-30 minutes
⚠ Test soup on a lab sensor first

SO updates can occasionally break customizations. Always test on a non-production sensor before rolling out to customer sensors.

Pushing CodeRed Config Updates to All Sensors

When you push changes to your CodeRed git repository, all sensors will automatically pull within 6 hours. To force immediate update:

# Force update on a single sensor
ssh -i NDR.pem ubuntu@<sensor-ip> "sudo /opt/codered/bin/codered-update.sh"

# Force update on multiple sensors
for ip in 10.0.1.10 10.0.1.11 10.0.1.12; do
    echo "Updating $ip..."
    ssh -i NDR.pem ubuntu@$ip "sudo /opt/codered/bin/codered-update.sh" &
done
wait
echo "All sensors updated."

6. Suricata Rules Management

Auto-Update Flow

Daily at 3:00 AM (±30 min jitter)
    │
    ├─ Download ET Open rules from rules.emergingthreats.net
    ├─ Validate: is gzip? rule count > 1000?
    ├─ Backup current rules (keep last 3 backups)
    ├─ Deploy new rules to /opt/so/saltstack/local/salt/suricata/rules/
    ├─ Reload Suricata (live rule reload — no traffic gap)
    └─ Log result to /var/log/codered/rule-update.log

Manual Rule Management

# Force immediate rule update
sudo /opt/codered/bin/update-rules.sh

# Check last update
cat /var/log/codered/last-rule-update.log
# Example output: 2026-03-18T03:14:22Z rules=52847

# View update history
cat /var/log/codered/rule-update.log

# View current rule count
grep -c "^alert\|^drop" /opt/so/saltstack/local/salt/suricata/rules/*.rules

# Check timer status
systemctl status codered-rule-update.timer
systemctl list-timers codered-rule-update

# Manually disable a noisy rule (add to threshold.config)
echo 'suppress gen_id 1, sig_id 2027757, track by_src, ip 10.0.0.0/8' \
    >> /opt/so/saltstack/local/salt/suricata/files/threshold.config
so-suricata-restart

Adding Custom Rules

# Create custom rules file
sudo vi /opt/so/saltstack/local/salt/suricata/rules/local.rules

# Example custom rules:
alert http any any -> any any (msg:"CUSTOM Suspicious User-Agent"; \
    http.user_agent; content:"evil-bot"; sid:9000001; rev:1;)

alert dns any any -> any any (msg:"CUSTOM Blocked Domain"; \
    dns.query; content:"malware-domain.com"; sid:9000002; rev:1;)

# Reload
so-suricata-restart

Adding Threat Intel Feeds

# Add IP blocklist to Suricata
# Download a threat intel IP list
curl -sL https://rules.emergingthreats.net/fwrules/emerging-Block-IPs.txt \
    | grep -v "^#" > /tmp/blocklist.txt

# Convert to Suricata rules
while read ip; do
    echo "drop ip $ip any -> any any (msg:\"THREAT-INTEL Blocked IP $ip\"; sid:9100001; rev:1;)"
done < /tmp/blocklist.txt >> /opt/so/saltstack/local/salt/suricata/rules/threat-intel.rules

so-suricata-restart

How Salt Works on the Sensor

# Salt runs in LOCAL mode (no master needed)
# All states are applied via:
salt-call --local state.apply codered

# Apply a specific sub-state:
salt-call --local state.apply codered.zeek
salt-call --local state.apply codered.hardening

# Test a state without applying (dry run):
salt-call --local state.apply codered.zeek test=True

# View pillar values:
salt-call --local pillar.items

# View grains:
salt-call --local grains.items

Customer Config ↔ Salt Pillar Flow

Customer CLI (cli.py)
    │
    ├─ Writes /etc/codered/sensor.conf
    │
    ▼
Salt Pillar (/opt/so/saltstack/local/pillar/codered/init.sls)
    │
    ├─ Reads sensor.conf via ini.get_ini()
    ├─ Exposes values as pillar['codered']['network']['monitor_interfaces'], etc.
    │
    ▼
Salt States
    │
    ├─ Use pillar values in Jinja templates
    ├─ Generate Zeek node.cfg, Suricata suricata.yaml, etc.
    └─ Apply configs + restart services

echo ""

cat /var/log/codered/last-rule-update.log 2>/dev/null || echo "Never"
echo ""
echo "=== Last Auto-Update ==="
cat /var/log/codered/last-update.log 2>/dev/null || echo "Never"
echo ""
echo "=== Suricata Stats ==="
docker exec so-suricata suricatasc -c dump-counters 2>/dev/null | grep -E "capture.kernel_packets|capture.kernel_drops" || echo "N/A"

11. Troubleshooting Playbook

Sensor Not Receiving Traffic

# 1. Check monitor interfaces are UP and PROMISC
ip -d link show ens34 | grep -i promisc

# 2. If not promisc, fix:
sudo ip link set ens34 up promisc on

# 3. Check if packets are arriving
sudo tcpdump -i ens34 -c 10

# 4. If no packets: problem is on switch/hypervisor side
#    - Verify SPAN session is active
#    - VMware: verify promiscuous mode on port group

# 5. If packets arrive but no Zeek logs:
docker logs so-zeek 2>&1 | tail -20
so-zeek-restart

Sensor Not Forwarding Logs

# 1. Check if logs exist locally
ls -la /nsm/zeek/logs/current/
ls -la /nsm/suricata/eve.json

# 2. Check forwarder is running
docker ps | grep -E "elastic-agent|filebeat"
# or
systemctl status filebeat vector

# 3. Test connectivity to CodeRed AI
curl -s -o /dev/null -w "%{http_code}" https://<codered-ai-ip>:9200
# Should return 200 or 401

# 4. Check forwarder logs
docker logs so-elastic-agent 2>&1 | tail -50
# or
journalctl -u filebeat -n 50

# 5. Verify firewall allows outbound
ufw status | grep 9200

High CPU / Memory

# 1. Identify the process
top -bn1 | head -20

# 2. If Zeek is using too much CPU:
# Reduce workers
sudo vi /etc/codered/sensor.conf
# Change: workers = 2 (instead of auto)
salt-call --local state.apply codered.zeek

# 3. If Suricata is using too much memory:
# Reduce stream memcap in suricata.yaml
# Default: 512mb, try 256mb for small sensors

# 4. If disk is full:
# Check what's using space
du -sh /nsm/* | sort -rh | head -10
# Clean old Zeek logs
find /nsm/zeek/logs -mtime +7 -delete
# Clean old PCAP
find /nsm/pcap -mtime +3 -delete

Customer Locked Out

# 1. SSH as admin
ssh -i NDR.pem ubuntu@<sensor-ip>

# 2. Reset coderedai password
echo "coderedai:coderedai" | sudo chpasswd

# 3. If customer changed password and forgot it:
echo "coderedai:NewTempPassword123" | sudo chpasswd
# Tell customer the temp password, they can change it via menu option 11

# 4. If SSH is completely broken:
# Access via hypervisor console (VMware/Proxmox)
# Login as ubuntu (if console login is available)
# Fix SSH: sudo systemctl restart sshd

Salt State Failure

# 1. Run Salt in debug mode
salt-call --local state.apply codered -l debug 2>&1 | tee /tmp/salt-debug.log

# 2. Check specific state
salt-call --local state.apply codered.zeek test=True

# 3. View pillar (check sensor.conf is parsed correctly)
salt-call --local pillar.items | grep -A5 codered

# 4. Common fixes:
# - sensor.conf has invalid INI syntax → fix and re-run
# - Salt state has Jinja error → check template syntax
# - Package install fails → check internet connectivity

Rule Update Failing

# 1. Check the log
cat /var/log/codered/rule-update.log

# 2. Common causes:
# - No internet: check DNS and HTTPS outbound
curl -sI https://rules.emergingthreats.net
# - Firewall blocking: check UFW
ufw status | grep 443
# - Disk full: check space
df -h /opt

# 3. Run manually with verbose output
bash -x /opt/codered/bin/update-rules.sh

#!/bin/bash
# Run as admin: sudo bash backup-sensor.sh
BACKUP_DIR="/tmp/codered-backup-$(hostname)-$(date +%Y%m%d)"
mkdir -p "$BACKUP_DIR"

cp /etc/codered/sensor.conf "$BACKUP_DIR/"
cp /opt/so/saltstack/local/salt/suricata/rules/local.rules "$BACKUP_DIR/" 2>/dev/null
cp /opt/so/saltstack/local/salt/suricata/files/threshold.config "$BACKUP_DIR/" 2>/dev/null
cp /home/ubuntu/.ssh/authorized_keys "$BACKUP_DIR/" 2>/dev/null
cp /opt/codered/VERSION "$BACKUP_DIR/"
hostname > "$BACKUP_DIR/hostname.txt"

tar czf "${BACKUP_DIR}.tar.gz" -C /tmp "$(basename $BACKUP_DIR)"
rm -rf "$BACKUP_DIR"
echo "Backup: ${BACKUP_DIR}.tar.gz"

# Download to your machine:
# scp -i NDR.pem ubuntu@sensor-ip:${BACKUP_DIR}.tar.gz .

Restore on New Sensor

# 1. Deploy fresh OVA
# 2. SSH as admin
ssh -i NDR.pem ubuntu@<new-sensor-ip>

# 3. Upload backup
scp -i NDR.pem codered-backup-*.tar.gz ubuntu@<new-sensor-ip>:/tmp/

# 4. Restore
sudo tar xzf /tmp/codered-backup-*.tar.gz -C /tmp/
sudo cp /tmp/codered-backup-*/sensor.conf /etc/codered/sensor.conf
sudo cp /tmp/codered-backup-*/local.rules /opt/so/saltstack/local/salt/suricata/rules/ 2>/dev/null
sudo cp /tmp/codered-backup-*/threshold.config /opt/so/saltstack/local/salt/suricata/files/ 2>/dev/null

# 5. Apply config
sudo salt-call --local state.apply codered

# 6. Mark setup complete (skip wizard)
sudo touch /etc/codered/.setup-complete

# Sensor is now a clone of the original

# Create a sensor list file

192.168.1.100

# Check status of all sensors
for ip in $(cat ~/sensors.txt); do
    echo "=== $ip ==="
    ssh -i NDR.pem -o ConnectTimeout=5 ubuntu@$ip "hostname; uptime -p; cat /opt/codered/VERSION" 2>/dev/null || echo "UNREACHABLE"
    echo ""
done

# Push rule update to all sensors
for ip in $(cat ~/sensors.txt); do
    echo "Updating rules on $ip..."
    ssh -i NDR.pem ubuntu@$ip "sudo /opt/codered/bin/update-rules.sh" &
done
wait

# Push config update to all sensors
for ip in $(cat ~/sensors.txt); do
    echo "Updating $ip..."
    ssh -i NDR.pem ubuntu@$ip "sudo /opt/codered/bin/codered-update.sh" &
done
wait

# Collect health from all sensors
for ip in $(cat ~/sensors.txt); do
    echo "=== $ip ==="
    ssh -i NDR.pem ubuntu@$ip "hostname; df -h / /nsm | tail -2; pgrep -c zeek; pgrep -c suricata" 2>/dev/null
    echo ""
done
