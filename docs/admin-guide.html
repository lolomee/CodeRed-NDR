1. Overview

Internal Document
This guide is for the CodeRed vendor team (developers, admins, support engineers) who build, maintain, and support the NDR sensor appliance. It is NOT for customers. A separate User Guide exists for customer-facing documentation.

What is CodeRed NDR?

CodeRed NDR is a network detection and response sensor distributed as a VMware OVA appliance. It passively monitors network traffic via SPAN/mirror ports and forwards security telemetry to the customer's SIEM.

2. Build Environment Setup

Requirements

  • OS: Ubuntu 24.04 LTS (also tested on 22.04)
  • Architecture: amd64 only
  • Minimum hardware: 2 CPU cores, 4 GB RAM, 20 GB disk
  • Recommended: 4+ CPU, 8+ GB RAM, 100+ GB disk
  • Internet access for package downloads and repo cloning

Clone the Repository

git clone https://github.com/lolomee/CodeRed-NDR.git ~/codered-sensor
cd ~/codered-sensor

Development Workflow

The recommended development workflow uses two environments:

  1. EC2 (AWS) for development and testing — Rapid iteration, SSH access, easy teardown. Run install.sh and test CLI/services here.
  2. VMware Workstation / ESXi for OVA export — Once tested, replicate the install on a clean VMware VM. Run prepare-ova.sh, shut down, and export the OVA.
Note
The prepare-ova.sh script removes AWS-specific artifacts (cloud-init, amazon-ssm-agent, EC2 metadata) so the OVA is clean. You can develop on EC2 but always do the final OVA export from VMware.

3. Installation Pipeline

Running the Installer

# From a local checkout:
sudo bash install.sh

# Or remote (curl-pipe):
curl -sSL https://raw.githubusercontent.com/lolomee/CodeRed-NDR/main/install.sh | sudo bash

The script is idempotent and safe to run multiple times. It detects existing installations and prompts before reinstalling.

Step-by-Step Breakdown

Pre-flight Checks

  • Must run as root
  • Validates Ubuntu 22.04 or 24.04, amd64 architecture
  • Checks minimum resources: 4 GB RAM, 2 CPU, 20 GB disk
  • If existing install found, prompts for reinstall/upgrade

Step 1: System Dependencies

Installs base packages via apt-get:

curl gnupg software-properties-common apt-transport-https
ca-certificates lsb-release python3 python3-configparser
dialog ethtool net-tools jq git tcpdump open-vm-tools logrotate

Step 2: Install Zeek

  • Adds OpenSUSE OBS APT repository for Zeek (security:/zeek/xUbuntu_<version>)
  • Installs the zeek package (installs to /opt/zeek/)
  • Falls back to 22.04 repo on 24.04 if primary fails
  • Adds /opt/zeek/bin to PATH via /etc/profile.d/zeek-path.sh
  • Disables the stock zeek systemd service (CodeRed manages its own)

Step 3: Install Suricata and Filebeat

  • Suricata: Adds ppa:oisf/suricata-stable, installs suricata + suricata-update. Enables community-id in suricata.yaml. Disables stock service.
  • Filebeat: Adds Elastic 8.x APT repo (artifacts.elastic.co/packages/8.x/apt), installs filebeat. Disables stock service.

Step 4: Deploy CodeRed NDR Files

Creates the directory structure and copies files from the repo:

DestinationContents
/opt/codered/shell/cli.pyManagement CLI
/opt/codered/bin/All management scripts (health-check, update-rules, etc.)
/opt/codered/firstboot/firstboot.shFirst-boot setup wizard
/opt/codered/VERSIONVersion string
/opt/codered/repo/Git clone for auto-updates
/etc/codered/codered.defaultsDefault configuration
/usr/local/bin/coderedndrCLI entry point wrapper
/nsm/zeek/logs/current/Zeek log output
/nsm/suricata/log/Suricata EVE JSON output

/var/log/codered/Audit, CLI, update, disk-cleanup logs

Also creates an inline update-rules.sh script (uses suricata-update) and an inline codered-update.sh if not present in the repo.

Step 5: Download Suricata Rules and GeoIP

  • Enables et/open and oisf/trafficid rule sources via suricata-update
  • Downloads GeoLite2-City.mmdb to /usr/share/GeoIP/

Step 6: Create Systemd Units and Set Permissions

Creates the following units:

  • codered-zeek.service — Zeek via zeekctl deploy/stop
  • codered-suricata.service — Suricata with config test pre-check
  • codered-firstboot.service — Runs wizard if /etc/codered/.setup-complete is missing
  • codered-rule-update.timer — Daily at 03:00 with 30 min jitter
  • codered-update.timer — Every 6 hours (15 min after boot, then every 6h)

Enables timers and firstboot service. Main services (zeek, suricata, filebeat) are NOT started by the installer — they start after the customer runs the setup wizard.

Permissions are set:

  • /opt/codered/ — root:root, 755
  • /opt/codered/bin/*.sh — 750
  • /etc/codered/ — root:root, 755; codered.defaults is 644
  • /var/log/codered/ — root:adm, 775
  • /nsm/ — root:root, 755

4. OVA Build Process

End-to-End Build Procedure

  1. Create a VMware VM: Ubuntu 24.04 LTS, 2+ NICs (NIC 1 = management, NIC 2 = monitor), 4+ CPU, 8+ GB RAM, 100+ GB disk (thin provision)
  2. Install Ubuntu Server (minimal), create coderedai user
  3. Clone the repo: git clone https://github.com/lolomee/CodeRed-NDR.git ~/codered-sensor
  4. Run the installer: sudo bash ~/codered-sensor/install.sh
  5. Verify services install correctly (do not start them)
  6. Run OVA prep: sudo bash ~/codered-sensor/bin/prepare-ova.sh
  7. Shut down the VM: sudo shutdown -h now
  8. Export as OVA from VMware Workstation (File > Export to OVF) or ESXi (Export OVF Template)

What prepare-ova.sh Does

This is a destructive, one-time script. It transforms a development VM into a distributable appliance.

Step 1: Create coderedndr User (Customer Account)

  • Creates coderedndr user with bash shell
  • Sets default password: CodeRed@NDR!
  • Forces password change on first login via chage -d 0
  • Configures CLI auto-launch in .bash_profile — when the customer SSHs in, the CLI starts automatically via sudo /usr/local/bin/coderedndr
  • Sets up restricted sudoers (see User Account Model)

Step 1b: Create cradmin User (Vendor Admin)

  • Creates cradmin user with full bash shell
  • No password set — account is locked, SSH key-only
  • Full sudo access (cradmin ALL=(ALL:ALL) ALL)
  • Prepares /home/cradmin/.ssh/authorized_keys
  • Custom .bashrc with aliases: ndr, health, logs-suri, logs-zeek, status
  • Red-colored prompt: [admin] codered-ndr:~$

Step 2: Lock Down Root

  • Locks root password (passwd -l root)
  • Sets PermitRootLogin no in SSH config
  • Restricts su to sudo group via pam_wheel.so
  • SSH AllowUsers: coderedndr cradmin
  • cradmin: key-only auth (PasswordAuthentication no, AuthenticationMethods publickey)

Step 3: Remove Unnecessary Packages

Purges: snapd, landscape, plymouth, unattended-upgrades, packagekit, fwupd, modemmanager, bluez, cups, avahi-daemon, whoopsie, apport, needrestart, and more. Reinstalls open-vm-tools afterward.

Step 4: Remove Cloud-Init and AWS Artifacts

Purges cloud-init, amazon-ssm-agent, aws-cli, ec2-instance-connect. Removes /etc/cloud, /var/lib/cloud, AWS log directories, and metadata caches.

Step 5: Remove Builder Account

Cannot delete coderedai while running as that user. Creates a systemd oneshot service (codered-remove-builder.service) that runs on next boot to delete the user and its home directory.

Step 6: Clean Filesystem

  • Removes SSH host keys (regenerated on first boot via regenerate-ssh-keys.service)
  • Clears bash history, Python history, vim info, caches for all users
  • Removes Claude Code session data
  • Truncates all logs, removes rotated/compressed logs, clears journal
  • Clears NDR data (/nsm/ logs, spool, extracted files)
  • Cleans APT cache, temp files, DHCP leases
  • Writes minimal default netplan (DHCP on all en*/eth*)
  • Clears /etc/machine-id (regenerates on boot for unique VM identity)
  • Removes persistent net rules, root SSH keys, git repos

Step 7a: Auto-Resize Disk Service

Creates /opt/codered/bin/auto-resize-disk.sh and codered-disk-resize.service. On every boot, the script uses growpart + resize2fs/xfs_growfs to automatically expand the partition and filesystem if the VMware admin increased the virtual disk size.

Step 7: Login Banner (MOTD)

Disables default Ubuntu MOTD. Creates /etc/update-motd.d/00-codered-banner that shows version, hostname, uptime, and service status on SSH login. Sets /etc/issue.net as SSH pre-authentication banner.

Step 8: Reset First-Boot State

Removes sensor.conf, .setup-complete, and .firstboot-complete markers. Resets Zeek node.cfg and Filebeat config to placeholder values. Stops and disables all NDR services so the customer starts fresh.

Step 9: Remove Builder Account

Removes coderedai from sudoers, revokes sudo group membership, schedules userdel -r on next boot.

Step 10: Final Disk Cleanup

Runs apt-get autoremove, cleans APT, zeroes free space with dd if=/dev/zero for better OVA compression.

Security Note
The sudoers configuration uses explicit command paths with no wildcards on sensitive commands. The --no-pager flag on journalctl prevents shell escape via pager programs. However, some entries use * wildcards on arguments (e.g., hostnamectl set-hostname *, ip link set * up). Review these for potential abuse vectors in high-security environments.

Adding Your SSH Key to cradmin

During OVA prep, the /home/cradmin/.ssh/authorized_keys file is created empty. To enable vendor support access:

# On the sensor (as cradmin or root):
echo "ssh-rsa AAAA... your-key" >> /home/cradmin/.ssh/authorized_keys

# Or remotely via the coderedndr account:
ssh coderedndr@sensor-ip
# Then from the CLI shell:
# (Exit CLI to get a shell, then)
sudo bash -c 'echo "ssh-rsa AAAA... your-key" >> /home/cradmin/.ssh/authorized_keys'
Tip
For production OVAs, add your team's SSH public keys to the prepare-ova.sh script before running it, so every shipped OVA has vendor access pre-configured.

6. CLI Architecture

Design

  • Language: Python 3 (no external pip dependencies)
  • Config: configparser for INI-style config files
  • System commands: subprocess.run() via the run_cmd() helper
  • Signals: SIGTSTP (Ctrl+Z) and SIGQUIT (Ctrl+\) are ignored to prevent shell escape

Entry Point

/usr/local/bin/coderedndr
  -> exec /usr/bin/python3 /opt/codered/shell/cli.py "$@"

Execution Flow

  1. main() sets up logging, records audit login event
  2. Checks is_configured() — looks for /etc/codered/.setup-complete
  3. If not configured: runs run_setup() (setup wizard)
  4. If configured: runs main_menu() (management menu loop)

Setup Wizard Flow

The wizard runs on first login when /etc/codered/.setup-complete does not exist:

  1. Hostname configuration
  2. Management network (interface, static/DHCP, IP/mask/GW/DNS)
  3. Monitor interfaces (multi-select, can skip)
  4. Sensor identity (name, registration token)
  5. SIEM log forwarding (address, port)
  6. Optional features (IPS mode)
  7. Confirmation and apply

Apply steps: save config, set hostname, configure management network, configure monitor interfaces, generate Zeek/Suricata/Filebeat configs, start services, update Suricata rules.

Config File Model

Two-layer configuration using Python configparser:

  1. /etc/codered/codered.defaults — shipped defaults, overwritten by updates
  2. /etc/codered/sensor.conf — customer-specific overrides, persists across updates

The load_config() function reads defaults first, then overlays sensor.conf. The save_config() function writes only to sensor.conf.

7. Detection Engine

Overview

CodeRed NDR includes custom Zeek detection scripts that provide behavioral detection beyond signature matching. These use Zeek's Notice and SumStats frameworks.

Scripts are installed to /opt/zeek/share/zeek/site/codered-detections/ and loaded via __load__.zeek.

Detection Scripts

beaconing.zeek — C2 Beaconing Detection

MITRE ATT&CK: T1071 (Application Layer Protocol), T1573 (Encrypted Channel)

Detects command-and-control beaconing by tracking connection intervals between source-destination pairs and flagging regular timing patterns.

How it works:

  • Tracks timestamps of all outbound connections per (src, dst) pair in a sliding window
  • Computes mean interval and standard deviation between consecutive connections
  • Calculates jitter ratio (coefficient of variation = stddev/mean)
  • Fires Beaconing_Detected notice if jitter is below threshold

Tunable thresholds:

ConstantDefaultDescription
beaconing_min_connections10Minimum connections in window to evaluate
beaconing_max_jitter0.15 (15%)Maximum jitter ratio to flag as beaconing
beaconing_window1 hrSliding window duration
beaconing_epoch1 hrSumStats epoch length

dns-anomaly.zeek — DGA & DNS Tunneling Detection

MITRE ATT&CK: T1568.002 (Domain Generation Algorithms), T1071.004 (DNS)

Three detection methods:

  1. DGA Detection: Computes Shannon entropy of the longest non-TLD label. High-entropy labels suggest algorithmically generated domains.
  2. DNS Tunneling (long labels): Flags DNS queries with subdomain labels exceeding a length threshold.
  3. DNS Tunneling (high rate): Uses SumStats to track query rate per (src, base_domain). Alerts when queries exceed threshold per minute.

Tunable thresholds:

ConstantDefaultDescription
dga_entropy_threshold3.5Shannon entropy threshold for DGA
dga_min_label_length8Minimum label length to evaluate
tunnel_max_label_length50Max label length before tunneling alert
tunnel_query_rate_threshold100.0Queries/min to single base domain
dns_rate_epoch1 minRate tracking epoch

long-connections.zeek — Long Connection & Exfiltration Detection

MITRE ATT&CK: T1041 (Exfiltration Over C2 Channel)

Two detection methods:

  1. Long Connection: Flags TCP connections lasting longer than the threshold. Excludes DNS, NTP, BGP by port and service name.
  2. Potential Exfiltration: Detects highly asymmetric data transfers (one side sent 10x+ more than the other) with minimum total bytes and minimum duration.
ConstantDefaultDescription
long_conn_threshold24 hrMax TCP duration before alert
exfil_min_bytes104857600 (100 MB)Minimum bytes to evaluate
exfil_asymmetry_ratio10.0Ratio threshold for asymmetric transfer
exfil_min_duration5 minMinimum connection duration

cert-anomaly.zeek — Certificate Anomaly Detection

MITRE ATT&CK: T1587.003 (Digital Certificates)

Detects on ssl_established event:

  • Self-signed certificates to external destinations (issuer == subject, dest is non-local)
  • Short-lived certificates (validity < threshold days)
  • Expired certificates (not_valid_after < network_time)
  • IP address as CN (regex match on Common Name field)
  • Empty subject
ConstantDefaultDescription
cert_min_validity_days7Certs valid less than this are flagged
cert_alert_self_signed_externalTAlert on self-signed to external
cert_alert_expiredTAlert on expired certs
cert_alert_ip_as_cnTAlert on IP address as CN
cert_suppress_interval1 hrSuppress interval per cert

scan-detect.zeek — Port Scan & Network Sweep Detection

MITRE ATT&CK: T1046 (Network Service Discovery)

Uses SumStats UNIQUE reducers:

  • Port Scan: Tracks unique destination ports per (src, dst) pair. Alerts when unique ports exceed threshold.
  • Network Sweep: Tracks unique destinations per (src, port). Alerts when unique hosts exceed threshold.

Observes both connection_attempt (SYN-only) and connection_established events.

ConstantDefaultDescription
scan_port_threshold20.0Unique ports to trigger port scan
sweep_host_threshold15.0Unique hosts to trigger sweep
scan_window5 minDetection window

How to Add New Detections

  1. Create a new .zeek file in zeek/codered-detections/
  2. Add @load ./your-detection to __load__.zeek
  3. Use the CodeRed module namespace
  4. Add your Notice types to Notice::Type enum
  5. Use &redef on thresholds so customers can tune via local.zeek
  6. Include MITRE ATT&CK reference in the notice message
  7. Test with zeek -r sample.pcap codered-detections

Tuning Thresholds

Customers can override any threshold in /opt/zeek/share/zeek/site/local.zeek:

@load codered-detections

# Tune beaconing sensitivity
redef CodeRed::beaconing_min_connections = 20;
redef CodeRed::beaconing_max_jitter = 0.10;

# Tune scan detection
redef CodeRed::scan_port_threshold = 50.0;
redef CodeRed::sweep_host_threshold = 25.0;

Suricata writes to /nsm/suricata/log/eve.json with these event types (configured in codered-override.yaml):

  • alert — IDS alerts with metadata and tagged packets
  • anomaly — Protocol anomalies
  • http — HTTP with extended fields
  • dns — DNS queries/responses
  • tls — TLS with JA3/JA4 fingerprints
  • files — File metadata with MD5/SHA256
  • smtp — SMTP with extended fields
  • ssh — SSH connections
  • flow — Flow records
  • netflow — NetFlow-style records

Filebeat Configuration

Filebeat is configured by apply_filebeat_config() in the CLI. It creates two inputs:

  1. Zeek logs: /nsm/zeek/logs/current/*.log (excludes .gz, stderr, stdout, capture_loss, reporter, stats)
  2. Suricata EVE: /nsm/suricata/log/eve.json (JSON parsed, keys under root)

ECS Processors:

processors:
  - add_host_metadata
  - add_fields:
      target: observer
      fields:
        name: "<sensor_name>"
        type: ndr
        vendor: CodeRed
        product: CodeRed NDR

Output options:

  • Elasticsearch: output.elasticsearch with http:// or https:// scheme
  • Logstash: output.logstash with optional TLS

Additionally, Filebeat module configs exist in the repo:

  • filebeat/zeek.yml — Zeek module with specific log paths for each log type (conn, dns, http, files, ssl, notice, smtp, ssh, ntp, dhcp, rdp, pe, x509, smb_files, smb_mapping)
  • filebeat/suricata.yml — Suricata module reading /nsm/suricata/log/eve.json

Disk Cleanup Behavior

The disk-cleanup.sh script runs periodically via codered-disk-cleanup.timer. It triggers when /nsm/ usage reaches 85%.

Cleanup phases (in priority order):

  1. Phase 1: Delete oldest 50 PCAP files (/nsm/pcap/*.pcap*)
  2. Phase 2: Delete oldest 100 archived Zeek logs (/nsm/zeek/logs/*.log.gz)
  3. Phase 3: Delete oldest 20 rotated Suricata logs (/nsm/suricata/log/eve.json.*)
  4. Phase 4: Delete oldest 200 extracted files (/nsm/zeek/extracted/)

Each phase re-checks disk usage before proceeding. All deletions are logged to /var/log/codered/disk-cleanup.log.

Rolling PCAP Capture

The start-pcap.sh script launches tcpdump with:

  • Rotates files hourly (-G 3600)
  • Splits at 100 MB per file (-C 100)
  • Compresses completed files (-z gzip)
  • Full packets (--snapshot-length=0)
  • Output: /nsm/pcap/<sensor_name>_YYYYMMDD_HHMMSS.pcap

10. Auto-Update Mechanism

How It Works

The codered-update.sh script runs every 6 hours via codered-update.timer. It:

  1. Changes to /opt/codered/repo/ (a git clone of the GitHub repo)
  2. Runs git pull --ff-only on the current branch
  3. Compares before/after commit hashes
  4. If changed, copies updated files to their installed locations:
Source (repo)DestinationNotes
shell/cli.py/opt/codered/shell/cli.pyRemoves chattr +i first, restores after
VERSION/opt/codered/VERSION
conf/codered.defaults/etc/codered/codered.defaults
bin/*/opt/codered/bin/All scripts
firstboot/firstboot.sh/opt/codered/firstboot/
Known Limitation
No signature verification. The auto-update mechanism pulls from the GitHub repo over HTTPS and copies files directly to /opt/codered/ without verifying GPG signatures, checksums, or code signing. A compromised GitHub account or MITM attack could push malicious code. Consider implementing signed releases for production deployments.
Known Limitation
What does NOT get updated: Zeek detection scripts (codered-detections/), Zeek intel feeds, Suricata rules, Filebeat module configs, systemd unit files, and sensor.conf are not updated by the auto-update mechanism. These must be updated manually or through their own update paths (rule timers, intel timers).

Update Interval

Default: every 6 hours with 30-minute randomized delay. Configured in codered-update.timer:

[Timer]
OnBootSec=15min
OnUnitActiveSec=6h
RandomizedDelaySec=30min
Persistent=true

How to Disable Auto-Update

# Disable the timer
sudo systemctl stop codered-update.timer
sudo systemctl disable codered-update.timer

# Or set in codered.defaults / sensor.conf:
[autoupdate]
enabled = no

Manual Update Procedure

# SSH in as cradmin
cd /opt/codered/repo
sudo git pull
sudo /opt/codered/bin/codered-update.sh

Logging

Update logs are written to /var/log/codered/update.log. The last successful update timestamp is stored in /var/log/codered/last-update.log.

11. Troubleshooting Guide

Common Issues and Fixes

Services won't start

Cause: Missing /etc/codered/sensor.conf or .setup-complete marker.

Fix: Run the setup wizard: sudo coderedndr (it auto-detects unconfigured state).

Cause: Monitor interface not found.

Fix: Check ip link show for available interfaces. Update via CLI menu option 7.

No logs being generated

Cause: Monitor interface not in promiscuous mode, or no SPAN traffic.

Fix:

# Check promisc mode
ip -d link show <interface> | grep PROMISC

# Manually set promisc
sudo ip link set <interface> up promisc on

# Verify SPAN is configured on the switch
# Check tcpdump can see traffic:
sudo tcpdump -i <interface> -c 10
Filebeat not forwarding to SIEM

Causes and fixes:

  • Wrong SIEM address: Reconfigure via CLI menu option 8.
  • TLS issues: If SIEM requires TLS, ensure siem_tls = true in config. Check /var/log/filebeat/filebeat for TLS errors.
  • Firewall blocking: Ensure sensor can reach SIEM on configured port (default 9200).
  • Filebeat errors: sudo tail -50 /var/log/filebeat/filebeat
Disk full (/nsm fills up)

Immediate fix:

sudo /opt/codered/bin/disk-cleanup.sh

Long-term:

  • Increase VM disk size in VMware (auto-resize on reboot handles the rest)
  • Reduce PCAP retention by adjusting start-pcap.sh
  • Check if codered-disk-cleanup.timer is enabled
High CPU usage

Cause: Traffic volume exceeds VM sizing.

Fix:

  • Check CPU per process: ps aux | grep -E 'zeek|suricata|filebeat'
  • Increase VM CPU cores
  • For Zeek: increase worker count in node.cfg (cluster mode for multi-core)
  • For Suricata: adjust threads setting
Zeek crashes or restarts frequently

Check:

# Check capture loss
cat /nsm/zeek/logs/current/capture_loss.log

# Check reporter for errors
cat /nsm/zeek/logs/current/reporter.log

# Check memory
ps aux | grep zeek

# Check journal
sudo journalctl -u codered-zeek --no-pager -n 100

Common causes: OOM (increase RAM), interface down, corrupt state (run zeekctl cleanup).

Suricata dropping packets 
# Check AF_PACKET stats
sudo suricatasc -c "iface-stat <monitor-interface>"

# Check suricata stats
sudo tail -100 /nsm/suricata/log/eve.json | jq 'select(.event_type=="stats")' | tail -1

Fix: increase VM resources, or tune Suricata AF_PACKET settings (ring-size, cluster-type).

13. Security Hardening

What prepare-ova.sh Hardens

  • Root account locked, SSH access disabled
  • su restricted to sudo group via pam_wheel.so
  • SSH: AllowUsers coderedndr cradmin, PermitRootLogin no
  • SSH key-only auth for cradmin (PasswordAuthentication no)
  • Login banner on SSH (/etc/issue.net)
  • Removed unnecessary services (snapd, avahi, cups, bluetooth, modemmanager, etc.)
  • Cloud-init and AWS artifacts removed
  • SSH host keys regenerated on first boot (unique per deployment)
  • Machine ID regenerated on boot (unique per VM)
  • Builder account deleted on first boot

Sudoers Design

FileUserAccess Level
/etc/sudoers.d/codered-ndr coderedndr Restricted NOPASSWD: specific systemctl commands, NDR scripts, network tools, ZeekControl, suricata-update. No shell escape vectors.
/etc/sudoers.d/codered-admin cradmin Full: ALL=(ALL:ALL) ALL
Security Note

The coderedndr sudoers uses --no-pager on journalctl to prevent pager shell escapes. However, the * wildcards on some commands (e.g., ip link set * up promisc on) could potentially be abused. In a high-security environment, consider restricting these further to specific interface names.

SSH Hardening

Configuration in /etc/ssh/sshd_config.d/90-codered-hardening.conf:

  • PermitRootLogin no
  • AllowUsers coderedndr cradmin
  • Banner /etc/issue.net
  • Match block for cradmin: PasswordAuthentication no, AuthenticationMethods publickey

CLI Immutability

The cli.py file has chattr +i set during auto-update, making it immutable (cannot be modified even by root without removing the attribute). This prevents accidental or malicious modification. The update script removes the flag before writing, then restores it.

Audit Trail

All CLI actions are logged to /var/log/codered/audit.log with:

  • UTC timestamp
  • Username
  • Source IP (from SSH_CLIENT environment variable)
  • Action performed
Known Limitation

Auto-update has no code signing. The update mechanism trusts the GitHub repository over HTTPS without additional verification. Consider implementing GPG-signed commits or release checksums.

Known Limitation

Default password is published. The OVA ships with the password CodeRed@NDR! for the coderedndr user. While chage -d 0 forces a change on first login, the default password is visible in the prepare-ova.sh script. Ensure customers change it immediately.

Known Limitation

TLS verification in Filebeat. When siem_tls = false, Filebeat communicates with the SIEM over plain HTTP. When TLS is enabled, the generated config uses verification_mode: certificate but no custom CA certificate path is configured. Customers using self-signed SIEM certificates will need to manually add the CA cert.

Restore

# List available backups
ls -la /var/lib/codered/backups/

# Restore a specific backup
cd /

# Restart services to pick up restored config
sudo systemctl restart codered-zeek codered-suricata filebeat

Factory Reset

sudo /opt/codered/bin/factory-reset.sh

Requires typing RESET to confirm. This will:

  • Stop and disable all NDR services
  • Delete sensor.conf, .setup-complete, .firstboot-complete
  • Reset Zeek node.cfg to placeholder
  • Clear all data in /nsm/ (logs, spool, extracted, pcap)
  • Truncate all CodeRed log files
  • Remove Filebeat config

After factory reset, the setup wizard runs on next login.

Version Upgrade Path

  1. Backup current config: sudo /opt/codered/bin/backup-config.sh
  2. Pull latest code: cd /opt/codered/repo && sudo git pull
  3. Run the update script: sudo /opt/codered/bin/codered-update.sh
  4. For major upgrades, re-run the installer: sudo bash /opt/codered/repo/install.sh
  5. Verify services: sudo /opt/codered/bin/health-check.sh

#fields	indicator	indicator_type	meta.source	meta.desc	meta.url

  1. Shut down the VM
  2. Increase the virtual disk size in VMware settings
  3. Power on the VM
  4. The auto-resize-disk.sh script automatically runs growpart + resize2fs
  5. Verify: df -h /
