1. Overview

The CodeRed NDR is a Network Detection and Response (NDR) appliance with built-in IDS (Intrusion Detection System) and optional IPS (Intrusion Prevention System) capabilities. It passively monitors your network traffic for threats, anomalies, and suspicious behavior using Zeek (behavioral metadata and UEBA) and Suricata (signature-based IDS/IPS engine with 50,000+ rules). In IDS mode, the sensor detects and alerts on malicious traffic. In IPS mode (inline deployment), it can actively block threats in real time. All alerts, network metadata, and detection events are forwarded to your central CodeRed AI platform for analysis and response.

How It Works

┌──────────────┐ │ Network │ │ Switch │ │ │ │ SPAN/Mirror ├──────┐ │ Port │ │ └──────────────┘ │ │ (mirrored traffic) ▼ ┌──────────────────────────────────────────────────────────────┐ │ CodeRed NDR │ │ │ │ ┌──────────┐ ┌───────────┐ ┌───────────────────┐ │ │ │ ens32 │ │ ens34 │ │ │ │ │ │ Mgmt │ │ Monitor │ │ Zeek + Suricata │ │ │ │ IP │ │ (SPAN) │───▶│ Analysis Engine │ │ │ └────┬─────┘ └───────────┘ │ │ │ │ │ └─────────┬─────────┘ │ │ │ │ │ │ │ ┌───────────────────────┘ │ │ │ ▼ │ │ │ ┌───────────────────┐ │ │ │ │ Log Forwarding │ │ │ │ │ (to CodeRed AI) │ │ │ │ └────────┬──────────┘ │ │ │ │ │ └────────┼─────────────┼───────────────────────────────────────┘ │ │ │ ▼ Management ┌──────────┐ Network │CodeRed AI│ └──────────┘

The sensor uses multiple network interfaces:

InterfacePurposeIP Address
NIC 1 (e.g., ens32)Management — SSH access, log forwarding to CodeRed AIStatic IP on your management network
NIC 2+ (e.g., ens34, ens35, ens36...)Monitoring — receives mirrored traffic from SPAN/mirror portsNo IP address (promiscuous mode)
💡 Multi-Zone Monitoring
You can add multiple monitoring interfaces to capture traffic from different network zones, VLANs, or switch segments simultaneously. Each monitoring NIC connects to a separate SPAN/mirror port. The sensor automatically distributes Zeek workers and Suricata capture threads across all monitoring interfaces.
⚠ IMPORTANT
Monitoring interfaces must be connected to SPAN/mirror ports. They do NOT need IP addresses — they listen passively in promiscuous mode.

Lateral Movement Detection

The sensor identifies attackers moving between hosts inside your network — one of the most critical and hard-to-detect attack phases.

ℹ How Lateral Movement Detection Works

The sensor tracks every internal connection in Zeek's conn.log with community-id correlation. When a host begins connecting to internal peers it has never contacted before — especially using administrative protocols (SMB, RDP, SSH, WMI) — the sensor flags this as potential lateral movement.

Encrypted Traffic Analysis (ETA)

Over 90% of internet traffic is encrypted. The sensor analyzes encrypted traffic without decryption using metadata and fingerprinting:

TechniqueWhat It Reveals
JA3 fingerprinting Unique TLS client fingerprint — identifies specific malware, C2 tools, and applications regardless of destination IP/domain
JA4+ fingerprinting Next-gen TLS fingerprinting with improved specificity, resistant to randomization
HASSH fingerprinting SSH client/server identification — detects unusual SSH tools and tunneling
Certificate analysis Self-signed certs, expired certs, newly issued certs, cert impersonation (e.g., "Microsoft" typosquatting)
TLS metadata SNI (Server Name Indication), cipher suites, TLS version downgrade attacks
💡 Why This Matters

Traditional IDS/IPS cannot inspect encrypted traffic content. The CodeRed NDR uses TLS metadata and fingerprinting to detect threats inside encrypted channels without requiring SSL/TLS decryption or certificate man-in-the-middle — preserving privacy while maintaining visibility.

4. Deployment Steps

Step 1: Import the OVA

VMware ESXi

  1. Log in to vSphere Client
  2. Right-click your host → Deploy OVF Template
  3. Select the codered-sensor-x.x.x.ova file
  4. Follow the wizard, accept defaults
  5. Before powering on, adjust CPU/RAM/disk if needed

Proxmox

  1. Copy the OVA to your Proxmox host
  2. Extract: tar xvf codered-sensor-x.x.x.ova
  3. Import: qm importdisk <vmid> codered-sensor.vmdk local-lvm
  4. Attach the disk and configure 2 NICs in the VM settings

VMware Workstation

  1. File → Open → select the .ova file
  2. Choose storage location, click Import
  3. Adjust CPU/RAM settings if needed

Step 2: Configure Virtual NICs

The sensor requires at least two virtual network adapters (add more for multi-zone monitoring):

AdapterConnect ToPurpose
NIC 1Your management networkSSH + CodeRed AI forwarding
NIC 2SPAN/mirror port group (zone 1)Receives mirrored traffic
NIC 3 (optional)SPAN/mirror port group (zone 2)Receives mirrored traffic from second network zone
NIC 4+ (optional)Additional SPAN/mirror port groupsAdditional network zones as needed
⚠ VMware Users
For all monitoring NICs, the port group must have Promiscuous Mode: Accept enabled under the Security settings. Without this, the sensor will not receive mirrored traffic.

Step 3: Power On and Configure

  1. Power on the VM
  2. SSH into the sensor: ssh coderedai@<sensor-ip>
    Default credentials: coderedai / coderedai
  3. Follow the first-time setup wizard (see Section 5)
  4. The sensor will start monitoring automatically after setup

5. Network Mirror / SPAN Configuration

The sensor needs to receive a copy of your network traffic. This is done by configuring a SPAN (Switched Port Analyzer) or mirror port on your network switch.

What Traffic to Mirror

Recommended SPAN sources (in priority order): ┌──────────────────────────────────────────┐ │ │ │ ★ Uplink to firewall/internet ◄───── Most important (north-south) │ ● Inter-VLAN trunk ports ◄───── East-west traffic │ ● Server farm switch ports ◄───── Critical assets │ ● DMZ switch ports ◄───── Exposed services │ │ └──────────────────────────────────────────┘
💡 TIP
At minimum, mirror your internet uplink (the port connecting to your firewall/router). This captures all north-south traffic entering and leaving your network.

Cisco IOS Switch

! Mirror the firewall uplink (Gi0/1) to sensor port (Gi0/24)
configure terminal
monitor session 1 source interface GigabitEthernet0/1 both
monitor session 1 destination interface GigabitEthernet0/24

! Or mirror an entire VLAN
monitor session 1 source vlan 10,20,30 both
monitor session 1 destination interface GigabitEthernet0/24

! Verify
show monitor session 1

Cisco Nexus (NX-OS)

configure terminal
monitor session 1 type span
  source interface Ethernet1/1 both
  destination interface Ethernet1/48
  no shut

Arista EOS

configure terminal
monitor session 1 source Ethernet1 both
monitor session 1 destination Ethernet48

Juniper EX / QFX

set forwarding-options analyzer SPAN input ingress interface ge-0/0/0.0
set forwarding-options analyzer SPAN input egress interface ge-0/0/0.0
set forwarding-options analyzer SPAN output interface ge-0/0/47.0

Meraki

Meraki has limited SPAN support. Go to Switch → Monitor → Packet Capture. For production, use a network TAP instead.

VMware vSwitch / Distributed vSwitch

Standard vSwitch

  1. vSphere Client → Host → Networking → vSwitch properties
  2. Select the port group for the sensor's monitoring NIC(s)
  3. Security tab → set Promiscuous Mode: Accept
  4. Connect the sensor's monitoring NICs to this port group (one per zone if needed)
  5. VMs on the same vSwitch will have their traffic visible to the sensor

Distributed vSwitch (VDS) — Recommended

  1. Create a new port group for SPAN (e.g., "SPAN-Destination")
  2. Go to VDS → Settings → Port Mirroring → New Session → Distributed Port Mirroring
  3. Select source port groups/uplinks to mirror
  4. Set destination to the sensor's port
  5. Enable Promiscuous Mode: Accept on the destination port group

Proxmox (Linux Bridge)

# On the Proxmox host, mirror traffic from vmbr0 to vmbr1

6. Cloud Deployment (AWS / Azure / Alibaba Cloud)

The CodeRed NDR can be deployed in public cloud environments to monitor cloud workloads, VPC traffic, and east-west communication between instances.

⚠ Key Difference: Cloud vs On-Premise

In cloud environments, there is no physical SPAN port. Instead, you use the cloud provider's traffic mirroring feature to copy VPC network traffic to the sensor's monitoring interface. Each cloud provider has a different implementation.

Architecture Overview

Amazon Web Services (AWS)

AWS provides VPC Traffic Mirroring to copy ENI (Elastic Network Interface) traffic to a mirror target.

Step 1: Launch the Sensor Instance

  1. Go to EC2 → Launch Instance
  2. Choose Ubuntu 22.04 / 24.04 AMI (or import the CodeRed OVA as a custom AMI)
  3. Instance type: c5.xlarge (4 vCPU, 8 GB) minimum, c5.2xlarge recommended
  4. Configure two or more network interfaces:
    • eth0 — Management (assign to your management subnet, with security group allowing SSH port 22 + CodeRed AI port 9200 outbound)
    • eth1 — Mirror target for zone 1 (assign to your monitoring subnet, security group allow all inbound from VPC)
    • eth2, eth3... (optional) — Additional mirror targets for other VPCs or subnets
  5. Storage: 100 GB gp3 minimum, 500 GB recommended
  6. Launch and SSH in to complete sensor setup

Step 2: Create Traffic Mirror Target

# Create a mirror target pointing to the sensor's eth1 ENI
aws ec2 create-traffic-mirror-target \
    --network-interface-id eni-0abc123def456789 \
    --description "CodeRed NDR"

# Note the TargetId from the output (e.g., tmt-0abc123456789)

Step 3: Create Traffic Mirror Filter

# Create a filter (mirror all traffic)
aws ec2 create-traffic-mirror-filter \
    --description "Mirror All Traffic"

# Add inbound rule (all traffic)
aws ec2 create-traffic-mirror-filter-rule \
    --traffic-mirror-filter-id tmf-0abc123456789 \
    --traffic-direction ingress \
    --rule-number 100 \
    --rule-action accept \
    --destination-cidr-block 0.0.0.0/0 \
    --source-cidr-block 0.0.0.0/0

# Add outbound rule (all traffic)
aws ec2 create-traffic-mirror-filter-rule \
    --traffic-mirror-filter-id tmf-0abc123456789 \
    --traffic-direction egress \
    --rule-number 100 \
    --rule-action accept \
    --destination-cidr-block 0.0.0.0/0 \
    --source-cidr-block 0.0.0.0/0

Step 4: Create Traffic Mirror Sessions

# For each instance you want to monitor, create a mirror session:
aws ec2 create-traffic-mirror-session \
    --network-interface-id eni-SOURCE-INSTANCE \
    --traffic-mirror-target-id tmt-0abc123456789 \
    --traffic-mirror-filter-id tmf-0abc123456789 \
    --session-number 1 \
    --description "Monitor Web Server"

# Repeat for each source instance
ℹ AWS Limitations
  • VPC Traffic Mirroring is supported on Nitro-based instances only (c5, m5, r5, t3, etc.)
  • Mirror target and source must be in the same VPC (or use a Gateway Load Balancer for cross-VPC)
  • Mirrored traffic is encapsulated in VXLAN (UDP port 4789) — the sensor handles decapsulation automatically
  • There is a per-session bandwidth cost for mirrored traffic

AWS Alternative: VPC Flow Logs + Sensor

If Traffic Mirroring is not available (e.g., non-Nitro instances), you can use VPC Flow Logs for metadata-only monitoring. This provides connection-level data (source/dest IP, port, protocol, bytes) but not full packet inspection.

Microsoft Azure

Azure provides Virtual Network TAP (vTAP) and Azure Network Watcher for traffic mirroring.

Step 1: Launch the Sensor VM

  1. Go to Azure Portal → Virtual Machines → Create
  2. Image: Ubuntu 22.04 / 24.04 (or upload the CodeRed VHD)
  3. Size: Standard_D4s_v3 (4 vCPU, 16 GB) minimum
  4. Networking: create the VM with two or more NICs:
    • NIC 1 — Management subnet (NSG: allow SSH 22 inbound, CodeRed AI port outbound)
    • NIC 2 — Monitor subnet zone 1 (NSG: allow all from VNet)
    • NIC 3+ (optional) — Additional monitor subnets for multi-zone
  5. Disk: 100 GB Premium SSD minimum
  6. Enable Accelerated Networking on both NICs for best performance

Step 2: Configure Virtual Network TAP

# Create a vTAP targeting the sensor's monitoring NIC(s)
az network vnet tap create \
    --resource-group myResourceGroup \
    --name codered-vtap \
    --destination nic-monitor-id \
    --port 4789

# Associate source NIC(s) with the vTAP
az network nic vtap-config create \
    --resource-group myResourceGroup \
    --nic-name source-vm-nic \
    --vnet-tap codered-vtap

Azure Alternative: NSG Flow Logs + Packet Capture

If vTAP is not available in your region:

  1. Enable NSG Flow Logs (v2) for metadata-level monitoring
  2. Use Network Watcher → Packet Capture for on-demand full packet capture
  3. Forward flow logs to the sensor via Azure Event Hub or storage account
ℹ Azure Limitations
  • Virtual Network TAP is in preview in some regions — check availability

Alibaba Cloud

Alibaba Cloud provides Traffic Mirroring for VPC traffic inspection.

Step 1: Launch the Sensor ECS Instance

  1. Go to ECS Console → Create Instance
  2. Image: Ubuntu 22.04 / 24.04 (or import the CodeRed image)
  3. Instance type: ecs.c6.xlarge (4 vCPU, 8 GB) minimum
  4. Configure two or more ENIs (Elastic Network Interfaces):
    • ENI 1 — Management (VSwitch in management subnet)
    • ENI 2 — Mirror target zone 1 (VSwitch in monitoring subnet, security group: allow all from VPC)
    • ENI 3+ (optional) — Additional mirror targets for other zones
  5. System disk: 100 GB ESSD minimum

Step 2: Configure Traffic Mirroring

  1. Go to VPC Console → Traffic Mirroring
  2. Create a Mirror Target: select the sensor's monitor ENI
  3. Create a Mirror Filter: configure to mirror all traffic (or specific protocols/ports)
  4. Create a Mirror Session: select source ENI(s) of instances to monitor, associate the target and filter
  5. Enable the session
# Using Alibaba Cloud CLI:
# Create mirror target
aliyun vpc CreateTrafficMirrorTarget \
    --RegionId cn-hangzhou \
    --TrafficMirrorTargetType NetworkInterface \
    --TrafficMirrorTargetId eni-SENSOR-MONITOR-ENI

# Create mirror filter (all traffic)
aliyun vpc CreateTrafficMirrorFilter \
    --RegionId cn-hangzhou \
    --TrafficMirrorFilterName "codered-filter"

# Create mirror session
aliyun vpc CreateTrafficMirrorSession \
    --RegionId cn-hangzhou \
    --TrafficMirrorSourceId eni-SOURCE-INSTANCE \
    --TrafficMirrorTargetId tmtarget-xxx \
    --TrafficMirrorFilterId tmfilter-xxx \
    --Priority 1 \
    --Enabled true
ℹ Alibaba Cloud Notes
  • Traffic Mirroring is available in most regions for VPC
  • Source and target must be in the same VPC
  • Supports mirroring from ECS, RDS, SLB, and other ENI-based services
  • Mirrored traffic uses VXLAN encapsulation

9. Troubleshooting

Sensor Is Not Receiving Traffic

  1. Check the monitor interfaces: Menu option 2 — verify all monitoring NICs show PROMISC
  2. Check SPAN on your switch: Verify each SPAN session is active (show monitor session 1 on Cisco)
  3. VMware users: Verify Promiscuous Mode: Accept on all monitoring port groups
  4. Check cable/link: Menu option 2 — verify all monitor interfaces show UP

Sensor Cannot Reach CodeRed AI

  1. Run diagnostics: Menu option 4 — check the CodeRed AI connectivity line
  2. Check firewall rules: Ensure port 9200 (or your configured port) is open from sensor to CodeRed AI
  3. Verify CodeRed AI is running: On your CodeRed AI server, confirm the service is listening on the correct port

High Disk Usage

  1. Menu option 1 (Status) — check /nsm disk usage
  2. If above 85%, the sensor will auto-clean old data
  3. Consider increasing the VM disk size in your hypervisor

Services Not Running

  1. Menu option 1 — check which services show as stopped
  2. Menu option 9 — restart all services or a specific one
  3. If services keep crashing, generate a support bundle (option 10) and contact support

Cannot SSH Into Sensor

  • Check the management IP via your hypervisor console
  • Verify NIC 1 is connected to the correct management network
  • Default credentials: coderedai / coderedai
  • If password was changed and forgotten, contact your administrator
