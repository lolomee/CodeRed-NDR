1. Introduction

What Is CodeRed NDR

CodeRed NDR is a network detection and response appliance that passively monitors your network traffic for threats, anomalies, and indicators of compromise. Delivered as a ready-to-deploy OVA virtual appliance, it combines best-of-breed open-source engines into a single, hardened sensor that forwards enriched security telemetry to your SIEM.

What It Detects

  • Signature-based threats — Thousands of Suricata IDS rules from Emerging Threats / Open detect known malware, exploits, and command-and-control traffic.
  • Behavioral anomalies — The Zeek detection engine identifies C2 beaconing, DGA domains, DNS tunneling, certificate anomalies, long-lived connections, port scanning, and network sweeps.
  • Threat intelligence matching — Real-time IOC matching against abuse.ch feeds (URLhaus, Feodo Tracker, SSL Blacklist) via the Zeek Intel framework.

Architecture Overview

The sensor runs three core engines on a single VM:

ComponentRole
ZeekDeep protocol analysis, metadata extraction, behavioral detection scripts, and threat intel matching. Produces structured logs for DNS, HTTP, TLS/SSL, SSH, SMTP, FTP, DHCP, NTP, SMB, RDP, Modbus, DNP3, x509, files, and more.
SuricataSignature-based IDS engine processing ET/Open rules. Produces EVE JSON alerts and protocol logs.
FilebeatShips all Zeek and Suricata logs to the customer SIEM in ECS (Elastic Common Schema) format.

What's Included

  • C2 beaconing detection (regular-interval connection analysis)
  • DGA domain detection (high-entropy DNS query identification)
  • DNS tunneling detection (long labels, high query volume)
  • Certificate anomaly detection (self-signed, short validity, expired, suspicious CN)
  • Port scanning and network sweep detection
  • Long-lived connection detection (>24h, asymmetric data transfer)
  • Threat intelligence from abuse.ch: URLhaus, Feodo Tracker, SSL Blacklist
  • JA3/JA4 TLS fingerprinting and HASSH SSH fingerprinting
  • Community ID for cross-tool correlation
  • File extraction with MD5 and SHA256 hashing

Sizing Note
Increase resources proportionally for throughput above 500 Mbps. For 1 Gbps sustained traffic, allocate at least 8 vCPU and 16 GB RAM. The /nsm partition stores Zeek logs and Suricata EVE output; size accordingly for your retention needs.

OVA Import — VMware ESXi

  1. Log in to the ESXi web UI or vSphere Client.
  2. Select Create / Register VM → Deploy a virtual machine from an OVF or OVA file.
  3. Upload the codered-ndr-2.0.0.ova file.
  4. Choose a datastore with sufficient space (100 GB minimum, 500 GB recommended).
  5. Configure networking:
    • NIC 1 → Management port group (has IP, SSH, SIEM forwarding)
    • NIC 2 → SPAN/mirror port group (receives mirrored traffic, no IP)
  6. Complete the wizard and power on the VM.

OVA Import — VMware Workstation

  1. Open VMware Workstation. Select File → Open.
  2. Browse to the codered-ndr-2.0.0.ova file and click Import.
  3. After import, open VM Settings and configure:
    • Network Adapter 1 → Bridged or NAT (management)
    • Network Adapter 2 → Custom/Host-only or Bridged to SPAN segment
  4. Power on and proceed to initial setup.

Configuration Summary & Apply

The wizard displays a summary of all settings and asks for confirmation. After you confirm, the sensor applies configuration in five steps:

  1. [1/5] Saves configuration to /etc/codered/sensor.conf
  2. [2/5] Sets the system hostname
  3. [3/5] Configures the management network interface
  4. [4/5] Configures monitor interfaces (promiscuous mode, offloads disabled)
  5. [5/5] Configures and starts Zeek, Suricata, and Filebeat; updates Suricata rules
Setup Complete
After the wizard finishes, the sensor is active and monitoring traffic. On subsequent SSH logins you will see the management menu instead of the setup wizard.

4. SPAN / Mirror Port Configuration

Why the Sensor Needs Mirrored Traffic

CodeRed NDR operates as a passive sensor. It does not sit inline in the traffic path. Instead, it requires a copy of network traffic delivered to its monitor interface(s). This copy is created by configuring a SPAN (Switched Port Analyzer) session or mirror port on your network switch, or by using a hardware network TAP.

At minimum, mirror the traffic on your internet uplink port (the port connecting to your firewall or router). For broader visibility, mirror traffic from core switch trunks or key network segments.

  ┌──────────────┐
  │    Switch     │
  │               │   SPAN / Mirror
  │  Uplink ──────┼──────────────────▶  Sensor NIC 2
  │  (to FW)      │   (copy of traffic)
  └──────────────┘

Switch Configuration Examples

Cisco IOS

Mirror traffic from source interface to the sensor port:

monitor session 1 source interface GigabitEthernet0/1 both
monitor session 1 destination interface GigabitEthernet0/24

To mirror an entire VLAN:

monitor session 1 source vlan 10 both
monitor session 1 destination interface GigabitEthernet0/24
Cisco Nexus (NX-OS) 
monitor session 1 type span
  source interface Ethernet1/1 both
  destination interface Ethernet1/48
  no shut
Arista EOS 
monitor session 1 source Ethernet1 both
monitor session 1 destination Ethernet48
Juniper Junos 
set forwarding-options analyzer SPAN input ingress interface ge-0/0/0
set forwarding-options analyzer SPAN input egress interface ge-0/0/0
set forwarding-options analyzer SPAN output interface ge-0/0/47
HP / Aruba 
mirror-port 48
interface 1 monitor

On Aruba CX:

mirror session 1
  source interface 1/1/1 both
  destination interface 1/1/48
MikroTik RouterOS 
/interface ethernet switch
set switch1 mirror-source=ether1 mirror-target=ether24

For CRS3xx series:

/interface ethernet switch rule
add ports=ether1 mirror=yes mirror-target=ether24

VMware Virtual Environment

To monitor VM-to-VM traffic on the same ESXi host:

  1. Create a dedicated port group (e.g., "SPAN-Destination").
  2. Edit port group security settings: set Promiscuous Mode to Accept.
  3. Connect the sensor's NIC 2 to this port group.
  4. For a vSphere Distributed Switch (VDS), use Port Mirroring to mirror source ports to the sensor's port.
Important
Promiscuous mode must be enabled on the port group connected to the sensor's monitor NIC. Without this setting, the hypervisor will filter traffic and the sensor will see only its own frames.

Using a Hardware TAP

A passive network TAP is more reliable than SPAN for production deployments, especially at speeds of 1 Gbps and above. TAPs do not drop packets under load and introduce zero latency.

  Firewall ──▶ [ TAP ] ◀── Core Switch
                  │
                  ▼
            Sensor NIC 2

Recommended TAP vendors: Garland Technology, Gigamon, Dualcomm.

Use a TAP when:

  • Monitoring links at 1 Gbps or above
  • Your SPAN port drops packets under load
  • Compliance requirements mandate fully passive monitoring

Best Practices: What Traffic to Mirror

  • Internet uplink (essential) — All traffic between your network and the internet. Captures external C2, data exfiltration, and malware callbacks.
  • Core switch trunk — East-west traffic between VLANs. Detects lateral movement.
  • DMZ segment — Traffic to/from externally-facing servers.
  • Critical server VLAN — Domain controllers, file servers, databases.
  • Mirror both directions (ingress and egress) for complete visibility.

6. Detection Capabilities

Signature-Based Detection (Suricata)

Suricata processes all monitored traffic against the Emerging Threats / Open (ET/Open) rule set. Rules are automatically updated every 6 hours via cron. Community ID is enabled for all events to support cross-tool correlation. Detection categories include:

  • Malware command-and-control
  • Exploit attempts (CVEs)
  • Trojan activity
  • Phishing and social engineering
  • Policy violations (TOR, P2P, crypto mining)
  • Lateral movement (EternalBlue, PsExec, etc.)

Behavioral Detection (Zeek Detection Engine)

DetectionDescriptionIndicators
C2 Beaconing Identifies hosts making connections at regular intervals to the same destination, characteristic of malware check-in behavior. Regular interval timing, consistent payload sizes, external destinations
DGA Domains Detects DNS queries for algorithmically-generated domain names used by malware to locate C2 infrastructure. High character entropy, unusual character distribution, non-dictionary patterns
DNS Tunneling Identifies data exfiltration or C2 communication encoded within DNS queries and responses. Long DNS labels, high query volume to single domain, TXT record abuse, base32/64 encoding patterns
Long-Lived Connections Flags connections persisting longer than 24 hours with asymmetric data transfer, which may indicate ongoing data exfiltration or persistent backdoors. Duration >24h, asymmetric byte counts
Certificate Anomalies Detects TLS certificates that exhibit suspicious characteristics often associated with C2 or man-in-the-middle attacks. Self-signed certificates, short validity periods, expired certificates, suspicious Common Names
Port Scanning Detects hosts performing horizontal port scans against a single target or vertical scans across multiple targets. High unique port count per source, SYN-only packets, sequential port ranges
Network Sweeping Detects reconnaissance activity where a host contacts many destinations on the same port. High unique destination count per source, single port targeting

Threat Intelligence

CodeRed NDR automatically ingests and matches against the following threat intelligence feeds from abuse.ch:

FeedDescriptionIOC Types
URLhausMalware distribution URLsURLs, domains
Feodo TrackerBotnet C2 servers (Dridex, Emotet, TrickBot, QakBot)IP addresses, ports

Feeds are processed through the Zeek Intel framework for real-time IOC matching against live traffic. Matches generate entries in intel.log, which are forwarded to your SIEM. Feeds are auto-updated every 4 hours.

Advanced Fingerprinting

  • JA3/JA4 TLS fingerprinting — Identifies client and server TLS implementations regardless of encryption, enabling detection of known-malicious TLS clients.
  • HASSH SSH fingerprinting — Identifies SSH client and server implementations, useful for detecting anomalous SSH tools.
  • Community ID — All logs include the Community ID flow hash (enabled in both Zeek and Suricata), allowing seamless cross-tool correlation.
  • File extraction and hashing — Extracted files are hashed with both MD5 and SHA256 for threat intel lookups and forensic analysis. Suricata uses forced magic detection for accurate file typing.

FieldValue

observer.typendr

Additionally, each event includes a fields.source field set to either zeek or suricata, and a fields.sensor_name field matching the configured sensor name.

Data Sources Forwarded

Zeek Logs

All logs from /nsm/zeek/logs/current/*.log are shipped (excluding internal files: stderr.log, stdout.log, capture_loss.log, reporter.log, stats.log, and compressed .gz archives):

  • conn.log — Connection summaries (5-tuple, duration, bytes, community ID)
  • dns.log — DNS queries and responses
  • http.log — HTTP requests and responses
  • ssl.log — TLS handshake details and JA3/JA4 hashes
  • files.log — File transfers with MD5/SHA256 hashes
  • notice.log — Zeek detection framework alerts (beaconing, DGA, scanning, etc.)
  • smtp.log — Email transaction metadata
  • ssh.log — SSH session metadata and HASSH fingerprints
  • ntp.log — NTP server usage
  • dhcp.log — DHCP lease events
  • rdp.log — RDP connection metadata
  • x509.log — Certificate details
  • smb_files.log, smb_mapping.log — SMB file and share operations
  • intel.log — Threat intelligence matches

Suricata EVE

All events from /nsm/suricata/log/eve.json (JSON parsed with keys under root):

  • alert — Signature-based IDS alerts with metadata and tagged packets
  • anomaly — Protocol anomalies detected by the stream engine
  • http — Extended HTTP logging
  • dns — DNS transaction logging
  • tls — Extended TLS logging with JA3/JA4
  • files — File transfers with forced magic detection and MD5/SHA256 hashes
  • smtp — Extended SMTP logging
  • ssh — SSH session events
  • flow — Flow records
  • netflow — NetFlow-style records

Changing SIEM Destination After Deployment

Use CLI menu option 8 (SIEM destination) to change the SIEM address and port at any time. The sensor will regenerate the Filebeat configuration and restart the Filebeat service automatically.

TLS Configuration

During the first-boot wizard, you can enable TLS encryption for SIEM forwarding. When TLS is enabled:

  • Elasticsearch output: Uses HTTPS protocol with SSL certificate verification.
  • Logstash output: Enables SSL with certificate verification mode.

To configure custom CA certificates for your SIEM, place your CA bundle on the sensor and update the Filebeat configuration accordingly.

Index Naming

When forwarding to Elasticsearch, logs are indexed under the pattern codered-ndr-*. Create an index pattern or data view in Kibana matching this pattern to browse sensor data.

8. Troubleshooting

No Traffic Detected

  1. Run CLI option 2 (Network interfaces) — confirm the monitor interface shows the PROMISC flag.
  2. Verify the SPAN session is active on your switch (e.g., show monitor session 1 on Cisco).
  3. For VMware: confirm the port group has Promiscuous Mode set to Accept.
  4. Check that the monitor NIC is connected to the correct port group or physical port.
  5. Run CLI option 4 (Diagnostics) — verify monitor interface shows "UP, PROMISC".

Services Not Running

  1. Run CLI option 1 (Sensor status) to see which services are stopped.
  2. Run CLI option 9 (Restart services) → select 1. All NDR services.
  3. Check system log via CLI option 3 (View logs) → 5. System log for error messages.
  4. Verify the sensor has sufficient memory and disk space via CLI option 4 (Diagnostics).

SIEM Not Receiving Logs

  1. Run CLI option 4 (Diagnostics) — check the SIEM connectivity line. If it shows UNREACHABLE:
    • Verify firewall rules allow the sensor to reach the SIEM on the configured port.
    • Confirm the SIEM address and port are correct via CLI option 8.
  2. Confirm Filebeat is running via CLI option 1 (Sensor status).
  3. View Filebeat logs at /var/log/filebeat/filebeat for connection errors.
  4. If using TLS, verify the SIEM's certificate is trusted or that verification mode is configured correctly.
  5. Restart Filebeat via CLI option 9 → 4. Filebeat.

High Disk Usage

  1. Run CLI option 1 (Sensor status) — check /nsm disk usage.
  2. The sensor automatically cleans up old logs when disk usage exceeds 85%.
  3. If disk is consistently full, increase the VM's virtual disk size in your hypervisor. The sensor auto-resizes partitions on boot.
  4. Check for excessive logging (e.g., extremely high traffic throughput) and consider upgrading to the recommended specifications.

Packet Loss

  • Packet loss typically indicates the sensor is under-resourced for the traffic volume.
  • Increase CPU and memory allocation (recommended: 8 vCPU, 16 GB RAM for 1 Gbps).
  • The sensor automatically tunes interface offloads (rx, tx, sg, TSO, GSO, GRO, LRO are all disabled on monitor interfaces) and uses af_packet for high-performance capture.
  • If using SPAN, ensure your switch can handle the mirrored traffic volume without dropping frames. Consider a hardware TAP for high-throughput links.

Cannot SSH to Sensor

  1. Verify the sensor VM is powered on and the management NIC has link.
  2. If using DHCP, check your DHCP server for the assigned lease.
  3. Connect to the VM console directly to verify the management IP address.
  4. Confirm your firewall allows inbound TCP port 22 to the sensor.
  5. If fail2ban has blocked your IP (after repeated failed logins), wait 10 minutes or access via VM console to unblock.

Generating a Support Bundle

Use CLI option 10 (Support bundle) to generate a compressed diagnostic archive. The bundle is saved to /tmp/codered-diag-<timestamp>.tar.gz and can be downloaded via SCP. Sensitive tokens are automatically redacted. Send this to CodeRed support for analysis.

Re-Running Setup

To re-run the full setup wizard and reconfigure all settings from scratch, use CLI option 15 (Re-run setup wizard). This does not require reimporting the OVA.

Updates are managed via cron and require outbound HTTPS (port 443) access. The auto-update configuration is defined in /etc/codered/codered.defaults under the [autoupdate] section (enabled by default, 6-hour interval).

Manual Updates via CLI

Suricata rules are automatically updated during initial setup via /opt/codered/bin/update-rules.sh. Threat intel feeds are updated by /opt/codered/bin/update-intel.sh. Both scripts can be triggered manually by the vendor support team if needed.

Disk Management

The sensor includes automatic disk management:

  • Automatic cleanup: When disk usage exceeds 85%, the oldest Zeek rotated logs and Suricata archives are automatically purged.
  • Auto-resize on boot: If you expand the VM's virtual disk in your hypervisor, the sensor detects the additional space on next boot and expands the partition and filesystem automatically. No manual intervention is required.
  • Log rotation: Zeek logs rotate hourly into timestamped directories. Suricata EVE logs rotate daily. Filebeat retains 7 log files with 0644 permissions.

PCAP Capture and Retention

By default, the sensor does not perform full packet capture (PCAP) to conserve disk space. Zeek and Suricata extract rich metadata from traffic without retaining raw packets. If full PCAP is required for forensic purposes, consult your CodeRed support representative for guidance on enabling packet capture.

Backup and Restore

The sensor configuration is stored in /etc/codered/sensor.conf. To back up the sensor configuration:

$ scp coderedndr@<sensor-ip>:/etc/codered/sensor.conf ./sensor-backup.conf

To restore, copy the configuration back and restart services via CLI option 9:

$ scp ./sensor-backup.conf coderedndr@<sensor-ip>:/tmp/sensor.conf
# Then on the sensor, the admin can place it in /etc/codered/ and restart services

Audit Logging

All CLI actions are logged to /var/log/codered/audit.log with the following fields:

  • Timestamp (UTC, ISO 8601 format)
  • Username of the operator
  • Source IP (from SSH_CLIENT environment variable)
  • Action performed (login, logout, view, reconfigure, restart, etc.)

Example audit log entries:

2026-03-25T14:30:00Z user=coderedndr src=10.0.1.100 action=login
2026-03-25T14:30:05Z user=coderedndr src=10.0.1.100 action=view:status

2026-03-25T14:35:00Z user=coderedndr src=10.0.1.100 action=logout

The audit log can be viewed through CLI option 3 (View logs) → 6. Audit log.
