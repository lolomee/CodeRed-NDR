1. Overview

CodeRed NDR is a passive network detection and response sensor. It receives a copy of your network traffic via a SPAN or TAP port, analyses it in real time using Zeek and Suricata, and forwards alerts to your SIEM via Filebeat. A machine learning engine builds per-host behavioral baselines and alerts on deviation.

ComponentRole
ZeekProtocol parser — runs all 20 behavioral detection scripts
SuricataSignature-based IDS, JA3/JA4 TLS fingerprinting
FilebeatShips all alerts and logs to your SIEM
codered-mlML behavioral engine — per-host anomaly detection via Isolation Forest

The sensor requires two network connections:

  • NIC 1 — Management: SSH access and SIEM forwarding. Needs an IP address.
  • NIC 2+ — Monitor: Receives mirrored/SPAN traffic. No IP address.
Detection timing
Rule-based and signature detection starts as soon as traffic arrives. The ML behavioral engine activates per host after 50 hours of baseline data.

2. Quick start

  1. Import the OVA into VMware or Proxmox.
  2. Add a second network adapter in VM settings, connected to your SPAN port group.
  3. Enable Promiscuous Mode on the SPAN port group.
    VMware: Edit port group → Security → Promiscuous Mode → Accept
  4. Power on the VM.
  5. SSH in: ssh coderedndr@<sensor-ip>
    Default password: CodeRed@NDR! — you must change this on first login.
  6. Complete the setup wizard (~5 minutes):
    • Hostname and sensor name (shown in SIEM alerts)
    • Management IP — static or DHCP. Gateway is pinged before saving.
    • Monitor interface — wizard runs a live 5-second SPAN traffic check and warns if nothing is seen.
    • SIEM address and port — wizard tests TCP connectivity before saving.
  7. Wizard starts all services automatically. First alert appears in your SIEM within minutes.
Most common deployment mistake
If conn.log is empty after 60 seconds, the SPAN port is not delivering traffic. Check your switch mirror config and VMware port group promiscuous mode before anything else.

4. VMware & cloud deployment

Cloud mode must be enabled or detections will not work
AWS, Alibaba Cloud, and Azure all deliver mirrored traffic wrapped in a VXLAN header (UDP/4789). If the sensor is set to on-prem mode, it receives the encapsulated packets but cannot parse the inner traffic — all Zeek detection scripts and Suricata signatures are blind. Always set Deployment mode → cloud in the wizard or via Menu → 9 when deploying in a cloud environment.

VMware ESXi / vSphere — on-prem mode

  1. Create a port group named SPAN-Destination.
  2. Edit port group → Security → Promiscuous Mode → Accept (required).
  3. Connect sensor NIC 2 to this port group.
  4. For VDS: use Port Mirroring to mirror source ports to the sensor port.
  5. Wizard: select on-prem mode (default).

AWS — cloud mode

  1. Deploy the sensor OVA as an EC2 instance (t3.large minimum recommended).
  2. Add a second ENI as the monitor interface in the same VPC.
  3. Security group on the monitor ENI: allow UDP/4789 inbound from the ENIs you want to monitor.
  4. AWS Console → VPC → Traffic Mirroring → Mirror Targets:
    Create a target pointing to the sensor monitor ENI.
  5. AWS Console → VPC → Traffic Mirroring → Mirror Sessions:
    Source = ENI to monitor, Target = sensor monitor ENI.
  6. Wizard: select cloud mode, VXLAN port 4789.
Supported EC2 instance families
AWS Traffic Mirroring works on Nitro-based instances: M5/M6/M7/M8, C5/C6/C7/C8, T3/T4, R5/R6/R7, and most current generation types. T2 and older Xen-based instances are not supported. Check the AWS documentation for the full list.

Alibaba Cloud — cloud mode

  1. Deploy the sensor as an ECS instance.
  2. Add a second ENI as the monitor interface in the same VPC.
  3. Security group: allow UDP/4789 inbound from source ENIs.
  4. VPC Console → Traffic Mirroring → Create mirror session:
    Source = ENI to monitor, Destination = sensor monitor ENI.
  5. Wizard: select cloud mode, VXLAN port 4789.

Azure — cloud mode

Use Azure vTAP (preview) or Network Watcher Packet Capture to mirror traffic to the sensor monitor NIC. Set wizard to cloud mode, VXLAN port 4789.

GCP — cloud mode

Use a Packet Mirroring policy to mirror subnet traffic to the sensor internal IP. Set wizard to cloud mode, VXLAN port 4789.

Toggling cloud mode after deployment

Use Menu → 9 (Deployment mode) to switch between on-prem and cloud at any time. The sensor saves the setting, reconfigures Zeek and Suricata, and restarts both services automatically.

codered> 9
  Current mode: ON-PREM
  Deployment mode (on-prem/cloud): cloud
  VXLAN port [4789]: 4789
  Apply cloud mode? Restart Zeek and Suricata. [Y/n]: Y
  Zeek: OK
  Suricata: OK
  VXLAN decapsulation is now active.

5. Detection capabilities

CodeRed NDR covers ~75 MITRE ATT&CK techniques across 20 Zeek behavioral detection scripts plus Suricata signatures. All rule-based detections are immediate — no warm-up required.

C2 & covert channels

beaconing T1071, T1573

Detects C2 beaconing by tracking connection intervals per src-dst pair and flagging low-jitter timing. Also detects east-west internal beaconing (compromised hosts pivoting internally).

ja3-fingerprint T1573.002

TLS fingerprinting via JA3 and JA3S. Detects Cobalt Strike (15 profiles), Metasploit, Sliver, Brute Ratel, Havoc, AsyncRAT, LockBit, BlackCat, and XMRig. JA3+JA3S pair matching gives highest-confidence C2 identification.

http-c2 T1090.004, T1568.003

Domain fronting (SNI vs Host header mismatch), 20+ Cobalt Strike malleable C2 URI patterns, fast-flux DNS (5+ IPs per domain in 5 min), DNS-over-HTTPS bypass, and LOLBin user-agents (certutil, bitsadmin, mshta).

dns-anomaly T1568.002, T1071.004

DGA domain detection using Shannon entropy scoring on DNS labels. DNS tunneling via long subdomain labels and high query rates to a single domain.

icmp-tunnel T1095, T1572

ICMP data tunneling — oversized payloads (>200 bytes), high ICMP rate (>60/min), and large data volumes via ICMP (>100 KB/min). Catches icmptunnel, ptunnel, and Hans.

protocol-anomaly T1571, T1572, T1090

Protocol-port mismatch (SSH on 443, HTTP on non-standard ports), Tor exit node connections, IPv6-in-IPv4 tunneling, and SNMP enumeration bursts.

Lateral movement

lateral-smb T1021.002, T1543.003, T1550.002

SMB admin share access (C$, ADMIN$), remote service installation via named pipes (svcctl, atsvc, lsarpc, samr), and authentication failure spikes consistent with Pass-the-Hash.

lateral-rdp T1021.001, T1110.001/003

RDP brute force (8+ failures to one target in 3 min), credential spray (5+ unique targets), and workstation-to-workstation RDP hops. Configurable jump server whitelist.

lateral-wmi T1047, T1021.003/006

WMI lateral movement via DCE/RPC port 135, DCOM UUID binding (8 known WMI interfaces), WinRM/PowerShell remoting (ports 5985/5986), and PsExec/PAExec named pipe detection.

hassh-ssh T1021.004, T1110

SSH client HASSH fingerprinting — detects Paramiko, Impacket, AsyncSSH, Dropbear, and mass scanners. Also SSH brute force, credential spray, and SSH on non-standard ports.

Credential attacks

kerberos-attacks T1558.001-004, T1087

Kerberoasting (RC4 TGS-REQ burst), AS-REP Roasting (pre-auth missing), Golden/Silver Ticket (abnormal ticket lifetime), and Kerberos principal enumeration.

credential-access T1557.001, T1187, T1087.002

LLMNR/NBT-NS poisoning (Responder detection), NTLM relay, LDAP AD reconnaissance (30+ unique queries in 2 min), high-value LDAP filter detection (AdminCount, SPN, LAPS), and SAMR/LSARPC enumeration.

Exfiltration & impact

ransomware T1486, T1490, T1048, T1105

SMB mass file write burst (100+ unique files in 2 min), 20 ransomware file extensions, shadow copy deletion via WMI, double-extortion exfil (500 MB+), and executable staging downloads.

insider-threat T1074, T1048.003, T1020, T1078

Internal data staging (1 GB+ in 5 min), mass SMB share access (10+ unique shares in 5 min), FTP upload exfil (50 MB+), bulk email (50+ RCPT TO in 5 min), and off-hours large external transfers.

long-connections T1041, T1048

TCP connections exceeding 24 hours and asymmetric data transfers (10:1 out:in ratio over 100 MB) — both indicators of C2 channels or staged exfiltration.

Commodity threats

cryptomining T1496, T1571

Stratum protocol on 9 mining ports, DNS queries to 30+ known pool domains, mining software user-agents (XMRig, ccminer, sgminer), and Stratum JSON detection in HTTP bodies.

scan-detect T1046

Port scan (20+ unique ports to one destination in 5 min) and network sweep (15+ unique destinations on the same port in 5 min).

cert-anomaly T1587.003

Self-signed certs to external hosts, short-validity certs (<7 days), expired certs in active use, and IP addresses as certificate Common Names — all common in malware C2 infrastructure.

ot-anomaly & cloud-threats

Industrial protocol attacks (Modbus, DNP3) and cloud-specific threats (IMDS abuse, AWS key exposure, cloud storage C2). See Sections 7 and 8.

6. ML behavioral engine

The codered-ml service learns what normal looks like for each internal host and alerts when behavior deviates significantly from that host’s own history. No GPU, no labeled training data, no external dependencies beyond scikit-learn.

How it works

  1. Reads conn.log, dns.log, and http.log every 60 seconds.
  2. Aggregates 9 features per host per hour into a SQLite baseline database.
  3. After 50 hours of data, trains an Isolation Forest model using each host’s own 7-day history.
  4. Each new hour is scored against the host’s model. Anomalies are written to /nsm/codered/ml-alerts.json and shipped to SIEM alongside Zeek and Suricata alerts.

Resource limits
The ML engine is bounded to 25% CPU and 512 MB RAM at Nice=10, so it never starves Zeek or Suricata. Typical usage is ~150–200 MB RAM.

Checking ML status

systemctl status codered-ml
tail -f /var/log/codered/ml-engine.log

# Check warm-up progress per host (need 50 rows before ML activates)
sqlite3 /var/lib/codered/ml-baseline.db \
  "SELECT host, COUNT(*) AS hours FROM host_features GROUP BY host ORDER BY 2 DESC LIMIT 20;"

7. OT / ICS monitoring

The ot-anomaly script covers industrial control system protocols. Zeek ships with Modbus and DNP3 analyzers built in — CodeRed’s detection layer runs on top.

ProtocolPortCoverage
Modbus TCP502/tcpUnauthorized coil/register writes, dangerous function codes
DNP320000/tcp+udpDirect Operate (FC3/4), Cold/Warm Restart (FC13/14)
EtherNet/IP44818/tcpAllen-Bradley PLC connections from unexpected sources
IEC 60870-5-1042404/tcpSubstation automation from unexpected sources
OPC-UA4840/tcpIndustrial data exchange
BACnet47808/udpBuilding automation

Detections

  • Unauthorized Modbus writes — write commands from hosts not in the engineering stations whitelist
  • DNP3 dangerous function codes — Direct Operate, Cold/Warm Restart, Authentication Error
  • IT-to-OT lateral movement — connections from IT subnet to OT subnet on OT protocol ports
  • OT reconnaissance — 5+ unique OT devices contacted in 3 minutes
  • Engineering station abuse — OT protocol commands from unknown hosts

Configuration (add to /opt/zeek/share/zeek/site/local.zeek)

redef CodeRed::ot_engineering_stations += { 10.100.1.10, 10.100.1.11 };
redef CodeRed::ot_subnets += { 192.168.100.0/24 };
redef CodeRed::it_subnets += { 10.0.0.0/8 };
Tune before deploying in OT environments
Configure ot_engineering_stations before going live. Without the whitelist, all write commands generate alerts.

12. Security hardening

All hardening is applied automatically by the installer. No manual steps required.

SSH (/etc/ssh/sshd_config.d/90-codered-hardening.conf)

  • Root login disabled
  • Maximum 4 authentication attempts before disconnect
  • 30-second login grace time
  • Idle session disconnect after 15 minutes
  • Modern cipher/kex/MAC algorithms only — no CBC, no SHA-1
  • Login banner shown to all connecting clients

fail2ban

4 failed SSH attempts within 5 minutes = 30-minute IP ban. Config: /etc/fail2ban/jail.d/codered-sshd.conf.

File permissions

PathModeReason
/etc/codered/sensor.conf640Contains SIEM address and registration token
/etc/filebeat/filebeat.yml600May contain SIEM credentials
/opt/zeek/etc/node.cfg640 root:zeekExposes monitor interface names
/nsm/750 root:admRaw network security data

Password policy (menu option 11)

  • Current password verified via PAM before any change is accepted
  • Minimum 12 characters, must include uppercase + digit + symbol
  • New password piped directly to chpasswd — never written to disk

Restricting SSH by source IP

13. Tuning & customisation

All detection thresholds can be overridden without editing detection scripts. Add redef statements to /opt/zeek/share/zeek/site/local.zeek, then restart Zeek via Menu → 9 → Restart Zeek.

# ── Beaconing ─────────────────────────────────────────────────
# Raise minimum connections threshold (fewer false positives)
redef CodeRed::beaconing_min_connections = 15;

# Disable east-west beaconing detection (very flat networks)
redef CodeRed::beaconing_detect_internal = F;

# ── Lateral movement ──────────────────────────────────────────
# Whitelist known RDP jump servers
redef CodeRed::rdp_jump_servers += { 10.0.1.50 };

# Whitelist known SSH jump servers
redef CodeRed::ssh_jump_servers += { 10.0.1.51 };

# ── Active Directory ──────────────────────────────────────────
# Known domain controllers (improves Kerberos accuracy)
redef CodeRed::known_dcs += { 10.0.1.10, 10.0.1.11 };

# ── OT / ICS ──────────────────────────────────────────────────
redef CodeRed::ot_engineering_stations += { 10.100.1.10 };
redef CodeRed::ot_subnets += { 192.168.100.0/24 };
redef CodeRed::it_subnets += { 10.0.0.0/8 };

# ── DNS-over-HTTPS ────────────────────────────────────────────
redef CodeRed::approved_doh_resolvers += { "dns.company.com" };

# ── Insider threat ────────────────────────────────────────────
# Raise data staging threshold (default: 1 GB)
redef CodeRed::staging_bytes_threshold = 2147483648;  # 2 GB

# Adjust business hours for off-hours detection (UTC)
redef CodeRed::business_hours_start = 8;
redef CodeRed::business_hours_end   = 19;

ML engine tuning (/opt/codered/ml/codered-ml.py)

ParameterDefaultEffect

14. Troubleshooting

No traffic in conn.log after 60 seconds

  1. Menu → 2 — monitor NIC must show PROMISC and UP.
  2. Verify switch SPAN config is sending traffic to the correct port.
  3. VMware: port group promiscuous mode must be Accept.
  4. Test: sudo tcpdump -i <monitor-if> -c 10 — no output = SPAN not working.

SIEM showing UNREACHABLE in diagnostics

  1. Menu → 8 — confirm SIEM address and port are correct.
  2. Check firewall allows sensor management IP → SIEM on the configured port.
  3. Test: nc -z -w5 <siem-ip> <port>

Services not running

  1. Menu → 9 → Restart all.
  2. Menu → 3 → System log for errors.
  3. sudo journalctl -u codered-zeek -n 50 --no-pager

High Suricata packet drop rate

  1. Menu → 4 shows drop percentage.
  2. Increase vCPU count in hypervisor.
  3. sudo /opt/codered/bin/tune-interface.sh <monitor-if>

High disk usage

  1. Menu → 1 — check /nsm usage percentage.
  2. Expand VM disk in hypervisor (sensor auto-resizes on next boot).
  3. sudo /opt/codered/bin/disk-cleanup.sh

ML engine not producing alerts

  1. Check service: systemctl status codered-ml
  2. Check warm-up — need 50 rows per host before ML activates:
    sqlite3 /var/lib/codered/ml-baseline.db \
  "SELECT host, COUNT(*) AS hours FROM host_features GROUP BY host ORDER BY 2 DESC LIMIT 20;"
  3. View log: sudo tail -f /var/log/codered/ml-engine.log

Too many ML false positives

  • Allow one full week for baselines to stabilise — the first week is noisiest.
  • Raise ANOMALY_THRESHOLD to -0.25 in /opt/codered/ml/codered-ml.py, then systemctl restart codered-ml.
